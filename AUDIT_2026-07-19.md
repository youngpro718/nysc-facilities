### Round 8 (2026-07-20): Operations page layout — calendar bloat + broken sticky tabs
User complaint: has to scroll past the calendar to reach the maintenance workbench, the
issues tab, even lighting — asked whether coverage/ballast/fixtures-out are wired correctly.

**Quantified the calendar problem**: Maintenance tab stacked a full month-grid calendar
(min-height ~700-900px, 7×6 cells) between the header and the two lists people actually use
(Scheduled Maintenance, Maintenance Issues). Measured live: the "Scheduled Maintenance" card
didn't start until **2020px down the page** on a 714px-tall viewport — three screens of
scrolling before reaching real content. Fixed by splitting Maintenance into "Workbench"
(default — DCAS handoff + Scheduled + Issues, zero calendar) and "Calendar" (opt-in),
mirroring the Issues/Coverage/Rooms pattern the Lighting tab already uses. Same measurement
after the fix: 1242px, and critically, **zero calendar** in the default path — the remaining
distance is the page header, building filter, stat strip, and workbench stat tiles, all
legitimate content.

**Root-caused why switching tabs required scrolling back to the top**: the top-level
Overview/Issues/Maintenance/Lighting tab bar isn't sticky, so from deep in Maintenance,
reaching Issues meant manually scrolling back up first. Made it sticky
(`sticky top-11 sm:top-14 z-20`) — but it silently didn't work. Root cause traced further:
Layout.tsx's page-shell div uses `overflow-x-hidden`; per the CSS spec, setting only one
overflow axis to a non-`visible` value forces the browser to compute the *other* axis as
`auto`, so this div was secretly `overflow-y: auto` too — establishing a scroll-container
formatting context it never intended. `position: sticky` descendants stick relative to their
nearest such container, not necessarily the real window scroll, and this phantom container's
own `scrollTop` never moves (its content doesn't overflow it — the actual document does the
scrolling) — so any sticky element anywhere in the app, on every page, was silently inert.
Fixed by swapping to `overflow-x-clip` (Tailwind, clips without creating a scroll container —
the standard fix for this exact gotcha). Verified live: sticky tab bar now holds at
`top: 56px` after scrolling 1500px down, and clicking "Issues" from that depth works
instantly with no back-scroll required. Also verified no horizontal-scroll regression on
mobile (Spaces, Term Sheet — both still `scrollWidth === viewportWidth`).

**Answered the "hooked up correctly?" questions with real data, not guesses**:
- *Fixtures out / ballast issues / needs electrician* (Lighting → Issues sub-tab): all
  correctly wired to the real `lighting_fixtures` table via `getOpenLightingIssues()`
  (`status != 'functional'`). Currently 14 fixtures tracked, all `functional`, zero ballast
  issues, zero needing an electrician — the queue is empty because there's genuinely nothing
  wrong right now, not because it's broken.
- *Coverage* (Lighting → Coverage sub-tab, "LED conversion progress"): also correctly wired
  (`room_lighting_profiles` joined against all rooms, denominator = every room, not just
  profiled ones — so the % is honest, not inflated) and transparently labeled ("N profiled ·
  M unprofiled"). The real number: **1 of 169 rooms** has a lighting profile filled in. This
  is a data-entry completeness gap, not a code bug — the feature works, almost nobody has
  used it yet.

Files touched: `MaintenanceTab.tsx` (Workbench/Calendar split), `Operations.tsx` (sticky tab
bar), `Layout.tsx` (overflow-x-clip — global fix, benefits any future sticky element app-wide).

### Round 7 (2026-07-20): "critical" priority was invisible everywhere — CRITICAL bug
User reported the Operations page's "Critical queue" showed nothing despite having a real
critical issue. Root cause: `issue_priority_enum` is `low/medium/high/critical` (critical
added in a past migration per project history), but every "critical" filter across the app
was written before that and only ever checked `priority === 'high'` — genuinely
critical-priority issues fell through everywhere. Confirmed live: both open issues
("Ceiling leak", "Door can't lock") are `priority: critical` and were invisible to all of
the following.

**Highest-impact finding**: the bug lived in the database, not just the frontend. The
`get_issue_stats()` Postgres function computed its `critical` key by filtering
`priority = 'high'` — this feeds BOTH the Operations "Critical queue" AND the admin
dashboard's "Active Issues" stat and its red "N critical alerts require immediate attention"
banner (`getSystemAlerts` in commandCenterService.ts). That banner had **never fired once**
for any critical-priority issue since the enum value existed. Fixed via migration 114
(applied live) — `critical` now counts `priority in ('high','critical')`. Verified live:
Command Center now shows "1 critical alert — 2 critical issues require immediate attention."

Swept the whole codebase for the same pattern (`priority === 'high'` without a matching
`'critical'` check) and fixed every live instance:
- `useAdminIssuesData.ts` — Operations "Critical queue" (the one the user saw)
- `get_issue_stats()` Postgres function — dashboard stat + critical-alert banner (DB migration)
- `CommandCenter.tsx` — red priority badge on Recent Activity rows
- `IssueTimelineView.tsx` — worse than just missing the badge: a critical issue fell through
  to the `default` case of the icon picker and rendered a ✓ checkmark (implying resolved) —
  fixed the icon and badge-variant switches
- `CurrentIssuesDisplay.tsx` — the room-edit wizard's "current issues for this room" badge
- `GlobalKPIStrip.tsx` — dead/unmounted component, fixed anyway to not be a landmine later

Left alone (checked, not bugs): `IssueStats.tsx`'s "High Priority" card is honestly labeled
(doesn't claim to be "critical"); `useCourtIssuesIntegration.ts`'s toast variant and
`ModuleCards.tsx` are dead code, never called/mounted; `create_chambers_transition` and
`notify_admins_pending_supply_approval` reference *different* priority enums entirely
(maintenance_schedules, staff_tasks, supply_requests) — not the same bug class.

### Round 6 (2026-07-20): dashboard photo dismiss/rotate — user request
User asked how the "latest issue photo" hero behaves with multiple open photo issues, since
it visibly showed different photos at different times and they wanted a way to say "I've seen
this enough" without it disappearing outright. Investigated the code: it's deterministic
(newest open issue-with-photo per building, sorted by `created_at`), NOT a timed rotation —
no dismiss mechanism existed (a `_onMarkAsSeen` prop was wired through but never called; it's
dead plumbing from an unrelated feature — the reporter's own `issues.seen` flag).

Built a real per-admin dismiss/rotate: new `dashboard_photo_dismissals` table (user_id,
issue_id, RLS own-rows-only; migration 113, applied live) — deliberately separate from
`issues.seen` since that column means something different (reporter-facing) and reusing it
would leak between features. `useDismissedIssuePhotos` hook (optimistic dismiss). BuildingsGrid
now computes hero = newest photo-issue THIS admin hasn't dismissed; everything else (dismissed
or just outranked) passes down as `otherPhotoIssues`. BuildingCard: "✕ Seen it" chip on the
hero (dismisses → next photo instantly promotes); small thumbnail row below the hero for every
other open photo (click opens that issue via the `issue_id` deep link from Round 4). Nothing
is ever fully hidden — dismissing only changes hero vs. thumbnail size, matching the user's
"something in between" ask. Verified live end-to-end with a temporary second test issue
(inserted, dismissed, confirmed rotation + thumbnail click opened the right issue, deleted —
cascade FK cleaned the dismissal row automatically, table empty afterward).

### Round 5 (2026-07-20): visual/UX sweep — Personnel dead-end, FAB collision, mobile overflow
- **Personnel (`/occupants`) was a genuine dead-end**: both "Room Access" and "Occupant
  Departure" required typing a raw database UUID into a text box — no admin could know a
  room's or occupant's internal ID from memory, so the page was functionally unusable.
  Replaced both with real search-and-select pickers: Room Access reuses the existing
  `DeliveryRoomPicker` (search by number/name/building); Occupant Departure gets a new
  `OccupantPicker` (search by name/department, backed by the `occupants` table — distinct
  from `profiles`/`personnel_profiles`). Also fixed the nav-vs-header title mismatch
  ("Personnel" vs "Access Management") and humanized the raw `work_location`/`primary_office`
  enum badge on the departure view.
- **Mobile FAB overlapped a real tap target**: the global FloatingActionButton (bottom-28
  right-4, shown on the admin root dashboard) sits exactly where BuildingCard's "View →"
  chip renders — since the FAB has higher z-index, a tap aimed at "View" opened the FAB's
  quick-action menu instead. The whole card is one giant button so nothing was truly
  unreachable, but that exact corner was hijacked. Fixed by hiding the decorative chip on
  mobile (`sm:flex`) — desktop is unaffected (no FAB there). Confirmed via `computer` click
  (not coordinate math, which was unreliable against the 2x screenshot scale) that FAB tap
  correctly opens Order Supplies/Make a Request/Request a Key/Report Issue.
- **Term Sheet toolbar overflowed on mobile**: five controls (Add Part, Notify everyone,
  Search, table/card toggle, Export) sat in one non-wrapping flex row — the search box was
  clipped off the right edge of the viewport below ~500px, effectively unreachable without
  horizontal scrolling nobody would think to do. Added `flex-wrap` so it wraps to two rows
  on narrow screens; unchanged on desktop.
- Swept for other FAB-collision candidates (grep for bottom-right absolute/fixed elements
  outside the FAB itself) — all other hits live on routes where `FloatingActionButton`
  already self-hides via `hiddenPaths` (`/spaces`, `/supplies`, `/operations`, etc.), so
  BuildingCard was the only real collision.
