# NYSC Facilities — Post-Soft-Release Audit (2026-06-11)

Follow-up to [AUDIT_2026-06-09.md](AUDIT_2026-06-09.md). Triggered by soft-release issues: login failures, rate-limit lockouts, "data error" messages, and supply requests not appearing for Court Aides. Fresh pull of `main` (e0345ddf, 120 commits since last audit), verified with typecheck, 45/45 unit tests, live DB inspection, and a logged-in walkthrough as the court-aide test user.

## Root causes of the soft-release issues (all fixed)

### 1. Rate-limit lockouts — every login click burned **2** attempts
`SecureForm.validateForm` called `check_rate_limit` (which **increments** the counter), then `useSecureAuth.secureSignIn` called it again. With the enforcement max of 5 (see #2), users were locked out after 2–3 real tries. Verified live: one submit → `app_rate_limits.attempts` jumped by 2.
**Fix:** removed the incrementing pre-check from `SecureForm`; the authoritative check lives in `secureSignIn`. Lockout errors thrown there now also set the form's rate-limited UI state.

### 2. Rate-limit enforcement and messaging disagreed (5 vs 10)
`checkRateLimit` defaulted to `RETRY.securityMaxAttempts` (**5**) while the error message computed "attempts remaining" from `RETRY.loginMaxAttempts` (**10**). Users were told "6 attempts remaining," then locked on the next try.
**Fix:** `secureSignIn` now passes `RETRY.loginMaxAttempts` explicitly.

### 3. Rate-limit buckets split by email case
`jduchate@` and `Jduchate@` were separate identifiers (seen live in `app_rate_limits`). Also a bypass vector (case-vary to get fresh buckets).
**Fix:** `secureSignIn`/`secureSignUp` normalize `email.trim().toLowerCase()` before rate-limit, reset, and auth calls.

### 4. Court Aide supply panel: query 400'd, error masked as "All caught up!"
`SupplyFulfillmentPanel` selected `supply_requests.notes` — **a column that doesn't exist** (the field is `description`). PostgREST rejected the whole query; the panel rendered `isError` with the friendly empty state, so the queue looked empty instead of broken. This is the "supplies not popping up" bug.
**Fixes:** select/display `description`; added a real error state with a Retry button, distinct from the empty state.

### 5. `approved` orders invisible to Court Aides
The panel filtered `['submitted','received','picking','ready']` — omitting `approved`, a mandatory step in the state machine (`pending_approval → approved → received → picking`). Admin-approved restricted orders vanished from the Work Center (they showed in Supply Room, which includes `approved`).
**Fix:** panel now includes `approved` (badge + "new order" grouping). Verified live: the stuck approved order appears.

### 6. RLS hid orders and requester names from fulfillment staff
- `supply_requests_read`/`update` allowed only `admin/cmc/court_aide` — the **`purchasing`** role (1 live user) could only see *their own* requests in the Supply Room. The DB status-transition trigger explicitly trusts `purchasing`/`facilities_manager`/`system_admin`, so policies and trigger disagreed.
- `profiles_read` allowed only self or `admin/cmc` — requester names rendered blank for aides/purchasing.
**Fixes (migrations):** `align_supply_request_rls_roles_with_workflow` (role lists now match the trigger across requests + items policies) and `allow_fulfillment_staff_to_read_profiles`.

### 7. Admin approval notifications silently never created
`submitSupplyOrder` inserted into `admin_notifications` client-side, but RLS only lets **admins** insert there — and `supabase.insert()` doesn't throw, so the failure was invisible. Admins were never notified of pending approvals.
**Fix (migration `notify_admins_on_pending_supply_approval`):** AFTER INSERT/UPDATE trigger on `supply_requests` creates the notification server-side; removed the dead client insert.

### 8. Work Center "Schedule" panel 400'd on a wrong FK hint
`TodaySchedule` embedded `profiles!staff_tasks_created_by_fkey` — that FK points to `auth.users`; the profiles FK is `staff_tasks_created_by_profiles_fkey`. Query failed; panel always showed "No scheduled tasks."
**Fix:** corrected the hint. Console now clean on `/work-center`.

## Term Sheet rollover rebuild (same session, later)

The monthly term rollover had never worked — the sheet was stuck on Term III (ended March 29) because of three stacked DB landmines, found by actually exercising the flow:

1. `court_terms.term_number` and `.location` are legacy NOT NULL columns with no defaults → every "Start Next Term" insert failed. (Migration `court_terms_safe_defaults` + client now supplies them.)
2. `court_assignments.room_number` NOT NULL but omitted by the `copy_term_assignments` RPC → copy failed. (Migration `fix_copy_term_assignments_room_number`.)
3. **`idx_court_assignments_room_id_unique`** — a unique index on `room_id` alone, predating multi-term support: a room could appear in only ONE assignment across ALL terms, making term copies structurally impossible. Replaced with UNIQUE `(term_id, room_id)` (migration `court_assignments_unique_per_term`).

New capabilities (admin + court_liaison edit; everyone views):
- **Start Next Term dialog**: suggested name (Term III → Term IV; handles roman + arabic numerals), date defaults that never back-date after a gap, overlap validation, optional copy-from-previous with verified row count.
- **Add Part** (room picker of active courtrooms) and **Remove Part** (confirm dialog) — previously rows could only be created/deleted via SQL.
- **Room is now editable** in the assignment dialog (was a disabled input) — parts can move courtrooms between terms.
- RLS aligned with the UI: `court_terms` + `court_admin_staff` policies extended to `court_liaison`; `copy_term_assignments` gated internally (was callable by any authenticated user). Staff-directory save now detects RLS-denied zero-row updates instead of toasting false success.
- **Term-scoped the six other `court_assignments` consumers** (`useCourtAssignmentsMap`, `TermSheetPreview`, `staffManagement`, `judgeManagement`, `useEnhancedRoomData`) via new `getCurrentTermId()` helper — with 2+ terms their unscoped `maybeSingle()` room lookups would error or touch past-term rows; judge-assignment inserts now carry `term_id`.

Verified live as admin: created **Term IV (June 11 – July 9, Active, 32 assignments copied)**, added part "QA TEST" in Room 687, removed it, browsed back to Term III (history intact). Console clean; tsc + 45/45 tests pass.
