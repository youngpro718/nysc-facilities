# User-Side Audit — 2026-06-20

Full walkthrough of the app as a logged-in **admin** (jduchatelier), desktop + mobile widths,
driven through the live dev server. Goal: confirm what works, what's broken, what's confusing,
and what to correct before moving forward.

**Method:** navigated every primary route, harvested page health (headings, blank/error states,
console errors), verified data against the Supabase schema, and traced root causes in code.
**No uncaught console errors were seen on any route during the walk.**

---

## ✅ Fixed & verified this session

### 1. Stock search "refreshed after every letter" + lost focus  *(reported)*
- **Root cause:** `InventoryItemsPanel` had `if (isLoading) return <Loading/>`. Every keystroke
  changed `searchQuery`, which is part of the React-Query key. A new key has no cached data, so
  `isLoading` flips true → the **whole panel (including the search box) unmounted** → the input
  lost focus → user had to click back in. There was also **no debounce**, so it fired one DB
  query per letter.
- **Fix:** debounce the term (300 ms) before it drives the query, and add
  `placeholderData: keepPreviousData` so previous results stay on screen during refetch and the
  panel never unmounts.
- **Verified:** typed "pen" character-by-character — the input stayed the *same DOM element* and
  kept focus through the debounce + refetch (`inputIsSameElement: true`, `activeElementStillSearch: true`).

### 2. Stock search returned nothing for names with parentheses  *(found during #1)*
- **Root cause:** the search used `.or("name.ilike.%term%,…")`. PostgREST treats `( ) ,` as
  grouping syntax, so searching `Batteries (AA)` broke the filter and returned **0 results** —
  and most item names contain parentheses.
- **Fix:** quote the filter value (`name.ilike."%term%"`) and strip stray quotes/backslashes.
- **Verified:** `Batteries` → 5 results, and `Batteries (AA)` → the exact item (was empty before).

### 3. "Unit" field showing the item name ("battery")  *(reported)*
- **Root cause:** *not* auto-fill. The data model is deliberate — `unit` = the **smallest
  countable item** ("battery", "pen", "sheet"), with `pack_size` (e.g. 4) and `packaging_note`
  ("1 case = 24 boxes × 8 packs") alongside. The DB columns exist and 6 items use them. But the
  Stock-tab editors (`EditItemDialog`, `CreateItemDialog`) **hid `pack_size` and `packaging_note`**
  and labeled the field just "Unit" — so "battery" looked wrong/orphaned.
- **Fix:** relabel **"Smallest unit"**, add **Pack size** + **Packaging note** inputs (saved to the
  existing columns), and a helper line: *"Quantity is counted in smallest units… use pack size and
  the packaging note to spell out how they're ordered — e.g. 4 batteries = 1 pack, 12 pens = 1 box,
  1 ream = 1 unit."* Stock cards now also show `· 4/pack`.
- **Verified:** editing Batteries (AA) shows Smallest unit `battery`, Pack size `4`,
  Packaging note `1 case = 24 boxes × 8 packs`.

### 4. 🔴 "My Requests" page was completely broken  *(found during audit — regression)*
- **Symptom:** `/my-requests` showed **"Could not load requests."** for everyone — no supply orders,
  no service requests.
- **Root cause:** `useMyRequests` runs three queries and throws on any error. **All three were broken:**
  1. queried the **`key_requests` table**, which was **dropped** in the 2026-06-14 key-request removal;
  2. selected **`supply_requests.display_id`**, a column that doesn't exist;
  3. selected **`staff_tasks.room_id`**, which doesn't exist (it's `from_room_id` / `to_room_id`).
- **Fix:** removed the `key_requests` query + realtime subscription, dropped the non-existent
  columns, mapped location to `from_room_id ?? to_room_id`, and cleaned the stale "key requests"
  copy + type filter on the page.
- **Verified:** page now lists real supply orders + service requests with live statuses; no console errors.

## 🟠 Open findings (not changed yet — need a decision)

### A. Order Supplies grid doesn't show unit / pack size  *(directly your concern)*
On the **Order Supplies** grid (`QuickOrderGrid`), item tiles show only *name · category · Add* —
**no unit and no pack size.** So when someone orders "1", there's nothing on screen telling them
whether that's 1 battery or 1 pack. The data exists (and a pack-equivalent shows inside `OrderCart`,
which prior QA flagged as a possibly-orphaned path), but the **primary ordering surface doesn't
surface it.**
**Recommend:** show `each · 4 per pack` (and the packaging note) on the tile and the quantity
stepper. This needs one product decision first: **are orders placed in singles or in packs?** The
data supports either, but the UI should declare it.

### B. "My Requests" shows raw room UUIDs
Service-request rows display the room **UUID** (e.g. `ea77794e-fd0f-…`) instead of the room name/number.
Functional, but confusing. **Recommend:** resolve `from_room_id` → "Room 1123 (ATI)".

### C. Key-request removal is incomplete in code (latent)
The 2026-06-14 key-request removal left orphaned code that still references the dropped
`key_requests` table: `keyRequestService.ts`, `useKeyRequests.ts`, `KeyRequestDialog.tsx`,
`KeyRequestsSection.tsx`, `useRequestActions.ts`, `useKeyOrders.ts`, `AdminKeyOrdersSection/Card`,
`KeyOrderSection.tsx`. These are **currently dormant** (not mounted/visible on `/keys`, which loads
fine), so nothing is user-facing-broken today — but it's dead code that will 400 if ever re-mounted.
**Recommend:** delete the orphaned key-request code.

### D. Bare `/admin` and `/users` bounce to the dashboard on a hard load/refresh
`/users` → `/admin` and a direct load of `/admin` (no `?tab=`) lands on the dashboard during the
auth-rehydration race. `/admin?tab=users` and `/admin?tab=system` work fine. Low impact (in-app
nav uses the tabbed URLs), but **deep-linking or refreshing the admin page drops you on the dashboard.**

### E. Inventory units mostly unset
**147 of 153** items have a blank `unit` (only the 5 battery types + Pens (Red) define unit + pack).
Now that the editor supports it, the rest can be filled in. This is business data (how many per
pack) — **you should set it**; happy to bulk-apply a list you provide.
