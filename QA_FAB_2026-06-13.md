# Deep FAB Audit — Mobile vs Desktop (2026-06-13)

In-depth audit of the Floating Action Button request system: Order Supplies, Set Up a Room, Request Help, Report Issue — mobile↔desktop behavior and the "gets stuck on rapid taps" bug. Five parallel agent tracks (FAB mechanics, Report Issue, Order Supplies, Setup/Help, mobile↔desktop parity) plus live reproduction in a 375px viewport and direct DB verification.

## How the FAB actually works
- `FloatingActionButton.tsx` renders **only on mobile** (`useIsMobile`, breakpoint **<768px**) and is **hidden for purchasing / court_aide / court_officer**. Single tap opens an inline action sheet (backdrop + framer-motion items).
- Actions: **Order Supplies** → `/request/supplies`; **Set Up a Room** → `/request/help?type=setup`; **Request Help** → `/request/help`; **Report Issue** → `QuickIssueReportButton` → `SimpleReportWizard` (a wizard, *not* the 745-LOC `ReportIssueDialog`, which is desktop-admin-only); **Request a Key** (non-admins) → `/my-requests?new=1`.
- Desktop has no FAB — standard users use the `UserDashboard` action buttons + the My Activity tabs.

## FIXED THIS SESSION (committed + pushed)

1. **FAB "stuck on rapid taps" (HIGH — reproduced).** Tapping faster than framer-motion's exit animation orphaned the full-screen backdrop (`fixed inset-0 bg-black/40`); the exiting node kept `pointer-events:auto`, so a dim overlay lingered and captured taps while the menu was logically closed — UI felt frozen. **Reproduced live**: during a 10-tap storm, samples showed the backdrop present + capturing while the FAB read "closed." Fix: `pointerEvents` is now animated via motion variants (`exit` → `'none'`) so a fading backdrop stops capturing immediately; added a 280ms toggle lock so taps can't out-pace the animation; `AnimatePresence initial={false}`, `type="button"`, actions via `closeAndRun`. **Re-verified: zero closed-but-capturing samples.** Commit `1247a21d`.
2. **Supply duplicate-order double-submit (HIGH).** A fast double-tap on "Submit Order" could fire `submitOrder` twice before `isSubmitting` disabled the button → two `supply_requests` rows. Added a synchronous `submittingRef` latch in `useOrderCart`. Also fixed `addItem` mutating a cart item in place. Commit `c7bf98ae`.
3. **Setup deep-link race + unhandled rejections.** `HelpRequestPage` now re-syncs step state when `?type` changes while mounted (ref-gated so the Back button isn't re-forced forward), making a future desktop "Set Up a Room" link work; both submit handlers wrapped in try/catch. Commit `c7bf98ae`.

## VERIFIED OK (not broken)
- **Report Issue insert is safe.** The wizard writes `title` (auto-generated), `description`, `issue_type`, `priority`, `status`, `location_description`, `seen`, `created_by`, `reported_by` — all confirmed present on the live `issues` table (the NOT-NULL ones are all supplied). The mobile flow is not silently failing.
- Setup quantity steppers use functional updaters (no stale-closure drops). Submit-error preserves the cart.

## STILL OPEN — needs your decision (design, not just code)

These are real gaps the agents found, but the fix is a product choice rather than a clear bug:

### Mobile↔desktop parity (HIGH)
- **A1. "Set Up a Room" is mobile-only.** Desktop standard users have no button to the structured event-setup form (the deep-link now works if something navigates to it — but nothing on desktop does). *Decision: add it to UserDashboard?*
- **A2. purchasing / court_aide / court_officer have NO request or report-issue entry point on EITHER platform** (no FAB on mobile; their nav has no Dashboard/My Activity). If a court officer needs to report a broken door or request a key, there's no path. *Decision: give these roles an entry point?*
- **A3. Resize = silent state loss.** `SupplyOrderPage` and `MyRequests` swap entire components on `isMobile` (768px). Crossing the breakpoint mid-order (tablet rotate / window resize) unmounts the in-progress cart/form and loses it. *Fix direction: render one responsive component instead of two.*

### Order Supplies — mobile vs desktop divergence (MEDIUM/HIGH)
- **A4. The best cart is orphaned.** `OrderCart` (required delivery-location validation, recent-locations chips, "needed by" date, high-quantity approval warning) is **never rendered** — the live path uses `OrderSummaryFooter` on both mobile and desktop, which **lets you submit an order with a blank delivery location** and gives **no warning when a large order silently routes to approval**. *Fix direction: port the required-location validation + approval warning to the live footer, delete the dead `OrderCart` branch.*
- **A5. Stepper last-write-wins.** Rapid +/- taps on supply quantities can drop an increment (handlers read `currentQty` from the render closure and set an absolute value). Lower severity; fix by switching to delta-based functional updates.

### Duplicate UIs for one intent (MEDIUM)
- **A6.** "Make a Request" has 3 implementations (FAB→/request/help, dashboard→/request/help, My Activity→`RequestTaskDialog`) and "Request a Key" has 2 (FAB→/my-requests vs dashboard inline `KeyRequestDialog`) — different forms/validation writing the same tables. *Fix direction: consolidate to shared components.*

### Report Issue polish (MEDIUM/LOW)
- **A7.** Submitting while a photo is still uploading drops that photo (the wizard's submit `disabled` doesn't include `PhotoCapture`'s `uploading` state).
- **A8.** `SimpleReportWizard`'s footer is `position:fixed` to the viewport but rendered inside a Drawer/Dialog — on the desktop dialog path the submit bar floats across the whole screen instead of anchoring to the dialog.
- **A9.** `usePhotoUpload` (the desktop `ReportIssueDialog` path) has a stale-closure bug dropping photos on concurrent uploads, no size/count/type limits; the desktop dialog also collects "Needs immediate attention" / "recurring" checkboxes it never saves.
- **A10.** Heavy report-issue forms are eagerly imported (not lazy); `QuickIssueReportButton` fires a `useOccupantAssignments` query on every mobile page even before the dialog opens.

Recommended next batch: **A1+A2 (entry-point parity)** and **A4 (supply delivery-location validation)** are the highest user-impact; **A3 (resize state loss)** is the highest correctness risk.
