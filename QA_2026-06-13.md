2. **`/assets` dead nav parent.** `navigation.tsx:87` emits a nav item `href:'/assets'` with no matching route → 404. Verified live.
3. **Fake analytics shown as real data.** `InventoryManagement.generateUsageTrends` (`InventoryManagement.tsx:191-199`) returns `Math.random()` for trend direction and a random 5–35% for "percentage change," rendered unconditionally under **"Usage Trends – Last 30 Days."** Reaches admin, purchasing, and court_aide via Supply Room → Inventory. ## SECURITY

Good news first: **every write path is properly RLS-gated server-side.** Verified policies — `key_requests` UPDATE = `is_privileged()`; `keys`/`rooms` writes = admin/privileged; `inventory_items` writes = admin/cmc/court_aide; `court_terms`/`court_assignments` = admin + court_liaison; `staff_tasks` correctly scoped. User role-change/delete/promote all go through SECURITY DEFINER RPCs. So a non-privileged user cannot modify keys, rooms, inventory, or approve requests even by calling the API directly.

4. **Route guards check a feature flag, not role (defense-in-depth gap).** `ModuleProtectedRoute` (`ModuleProtectedRoute.tsx:30`) gates `/spaces`, `/inventory`, `/keys`, `/operations`, `/supply-room` on `useEnabledModules[key]` — which defaults all-`true` for everyone — **not** on role. The correct role→module matrix `hasModuleAccess()` (`roleBasedRouting.ts:78-99`) is written but **imported nowhere (dead code)**. Net effect: a non-privileged user who types `/keys` or `/inventory` directly gets the page (it's only hidden from their nav).
   - **Actual data exposure is limited but real:** `keys`, `rooms`, `inventory_items`, `lighting_fixtures` all have `SELECT USING (true)` for authenticated — so a direct-navigating standard user can *read* the key list, room list, and inventory catalog. The *sensitive* relational data is safe: `key_assignments` (who holds which key) and `occupant_room_assignments` (who sits where) are restricted to self-or-privileged. So: catalog read-exposure + ungated UI, not a breach of sensitive records. Fix by wiring `hasModuleAccess` into the route guard.
5. **`admin` and `system_admin` are collapsed.** `ProtectedRoute.tsx:46` sets `requireSystemAdmin` from `isAdmin` (admin OR system_admin), so any `admin` reaches system-admin-only routes (`/admin/routing-rules`, `/admin/form-templates`). May be intentional — confirm whether the two roles are meant to differ.
6. **MFA enforcement is commented out** (`OnboardingGuard.tsx:146-157`) while the file's own doc says privileged roles "must have MFA." Either enable it or fix the comment so the posture isn't overstated. Email-verification is enforced in three different layers on three different fields — works today but fragile.
7. **PII in logs:** `SupplyRoom.tsx:23` `logger.debug` dumps the full profile object every render. Leaked-password protection (HaveIBeenPwned) still off (dashboard toggle).

*(Carried-forward, unchanged: ~204 `authenticated`-executable SECURITY DEFINER functions remain a large surface but most have internal `is_admin()` checks; the 4 `anon`-executable functions are the intended pre-auth login validators — not a finding.)*

## PERFORMANCE

8. **jsPDF eager-import regression (385 KB).** `services/forms/pdfGenerationService.ts:1` static-imports `jspdf`, pulled into the public Forms route via `PublicForms.tsx`/`FormTemplates.tsx`. Opening Forms downloads the full 385 KB before any PDF is requested. (TermSheet does this right with `await import`.) Fix: lazy-import jspdf at the call site.
9. **Zero `React.memo` in the entire codebase.** Big list pages (Inventory panels 500–650 LOC, Keys, supply cards) re-render every row on any parent state change. Memoize list-row/card components.
10. **Largest chunks:** main entry 563 KB / 177 KB gz (over Vite's warn limit), `vendor-charts` (recharts) 384 KB / 106 KB gz. Recharts is isolated but monolithic.
11. **Unbounded reads:** only 4 `.range()` across 847 `.from()` queries; many admin/room-history lists use `.select('*')` with no pagination. Two per-row mutation loops (`useRoomAssignment.ts:100`, `OccupantDepartureView.tsx:50`) should batch via `Promise.all`/`.in()`.
12. **Count-query churn:** the admin dashboard fires a burst of `HEAD` count queries that get ERR_ABORTED on refetch/unmount — benign but noisy; realtime invalidations may be over-firing.

## PER-ROLE UX

- **court_aide** *(best-built surface)*: Work Center loads clean. But supply fulfillment is **duplicated** — the Work Center panel and the full `/supply-room` dashboard both fulfill the same orders with different UIs, and the Work Center "Start" jumps `submitted→picking`, skipping `received`. `WorkCenterStats` shows `0/0/0/0` with no skeleton on slow loads (reads as "nothing to do").
- **purchasing**: nav includes **"Tasks"** (the generic court-aide staff-task board) which doesn't fit a supply-chain role. Main page `ImprovedSupplyStaffDashboard.tsx` has `@ts-nocheck` (type safety off). Inherits the fake Usage Trends.
- **court_officer**: **too thin** — nav is a single "Keys" item, the profile avatar is hidden for this role (`Layout.tsx:222`), and `/profile` isn't in their routes. They can't view their profile, change theme, or see notifications. Decide keys-only intent or restore self-service.
- **court_liaison**: has `operations` + `supply_requests` permissions but **no in-nav entry point** to report an issue or request supplies; their "Supply Room" nav item lands on the *fulfiller's* dashboard, not a request flow.
- **standard**: two different "Request a Key" entry points (dashboard inline dialog vs FAB → `/my-requests?new=1`); the FAB is **mobile-only**, so desktop standard users never get "Set Up a Room"/"Report Issue" as primary actions. Receipt "PDF download" and "Email" buttons are `coming soon` stubs (`ReceiptDialog.tsx:18,22`).
